RLSC v2.0

다운스트림 기획이 확장되면서 RLSC에 없는 정보가 드러났고,
Rule Linter로는 더 이상 대응할 수 없는 지경에 이르렀습니다.
정의를 확장하고, 정책을 각자 정의하며, 충돌 검사를 자동화하는 시스템을 제안합니다.

주관: 강서림 · 참여: MORDOR, AIP, AIR/내용시각화, Engine · Jira: MOR-1141

WHY NOW

왜 지금 v2인가

v1 정의 → 다운스트림 기획 확장 → RLSC에 없는 정보 발견 → Linter로 대응 한계.

▍v1 정의 이후 벌어진 일

다운스트림 기획이 RLSC를 넘어섬

RLSC v1에서 Role + Layout을 정의했지만,
다운스트림 스쿼드(AIP, AIR, 검색, 대치, 스마트블록)에서 기획 진행 중 사전 정의된 RLSC에 없는 정보가 필요해지거나, 별도 개념으로 기획을 진행하게 됨.

Rule Linter의 한계에 도달

변경/신규 요구사항을 Linter rule로 대응하려 했으나, rule-based로는 한계가 있음.
- Grouping 오류(G-1~G-12)가 reject의 77.1%를 차지.
- Decoration/Background 경계 같은 정의 자체의 문제는 rule-based로 해결 불가.

▍Top-down도 안 되고, 각자 개발도 문제 → Policy-Driven

기존 방식의 한계

Top-down: 프롬프트에 정의를 하드코딩하는 기존 방식은 이미 성공하지 못함.
MORDOR의 정의를 모든 스쿼드가 따를 수 있을지 보장되지 않고, 신규 정의 대응 속도 문제로 MORDOR가 병목이 될 우려.

각 스쿼드 입장에서도 상위 기획을 기다려야 하면, 자유롭게 정책을 마련할 수 없어 불편.

→ Bottom-up Policy 자율 정의

각 스쿼드가 자기 domain의 policy를 직접 정의.
MORDOR는 정의를 강제하는 대신 플랫폼만 제공:
policy 편집 CMS + 충돌 검사 자동화 시스템.

Top-down이면 정의자가 모든 사용처 기획을 직접 파악해야 하는데, 현실적으로 불가능.
→ 각자 정의하고, 충돌은 시스템이 자동 감지.

정의 확장

부족한 정보를 어떻게 표현할 것인가 - labels

.labels attribute가 제안되었으나, 모든 정보를 labels에 담을지, 기존 속성을 확장할지는 논의가 필요합니다.

참고: labels에 모든 것을 넣을지, 기존 속성(Role, LC.type)을 확장할지, 별도 개념으로 관리할지는 미결정 상태입니다.
2/5 회의: "labels에 모든 것을 몰아넣기보다 구분이 필요할 수 있음" · 2/23 회의: "structural, rhetorical, visualunit + named entity로 category화"
→ 구체적인 설계는 추후 결정 예정

.labels 구조 제안

설계 전제 : 각 축의 값은 배열로 복수 할당 가능. 하나의 요소가 맥락에 따라 여러 label을 동시에 가질 수 있음.

단일 label - 3개 축으로 레이블링 가능

  "visual_layout": ["Timeline"],
  "rhetorical": ["Problem"]
} }

multi-label — 하나의 요소가 여러 label 보유 가능

// LayoutContainer가 동시에 두 역할

  "structural": [
    "PageTitle",
    "ComponentTitle"
  ]

왜 전체 Policy를 하나의 context에?
Domain별 policy가 독립적으로 보이지만, 실제 정확한 RLSC 추론을 위해서는 Layout과 Role, Label이 서로의 정의를 참조할 수 있어야 합니다.
따라서 모든 domain의 작업 instruction이 동시에 주입된 상태에서 추론이 진행되어야 합니다.
