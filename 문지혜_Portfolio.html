문제 정의

디자인 템플릿(약 14만 페이지)에서 RLSC(Region·Layout·Structure·Component) 디자인 온톨로지를 LLM으로 추출해야 했습니다. 기존 단일 추론은 환각·구조 오류가 잦아, Generator의 결과를 Evaluator가 평가하고 그 피드백을 다시 LLM에게 전달해 재추론시키는 Agentic 구조가 필요했습니다.

나의 역할 (RLSC Generator + 파이프라인 오너)

  • RLSC Generator 구현 — 모듈 가장 앞단에서 LLM 호출·입출력 형식 설계. 주요 모듈 중 가장 큰 모듈을 담당.
  • 프롬프트·피드백 설계 — RLSC 생성용 + Evaluator 평가 결과를 반영하는 피드백 프롬프트 설계·반복 개선.
  • Evaluator 연동 — 평가 결과를 받아 피드백 프롬프트를 동적으로 구성, LLM에 재추론 요청.
  • Langfuse 관측·비용 트래킹 — 모든 LLM 호출에 trace_id를 부여해 비용·실패 원인을 추적 가능하게 함.
  • 전체 추출 파이프라인 — 3단계(JSON → Agentic RLSC 배치) + 4단계(Design Object 추출) 파이프라인을 pg-boss 큐·워커·REST API·백오피스 UI(Next.js)로 구현.

아키텍처 (3단계 배치 흐름)

[Frontend] AgenticRLSCDialog → POST /api/agentic-total-batch-conversion/agentic-rlsc/start [API] layouts 조건 조회 (json_sheet_data IS NOT NULL, agentic_jsonb IS NULL) → batchId(UUID) 생성, layoutId × jobCountPerLayout payload → pg-boss 큐에 비동기 send() / job_history upsert(status=pending) [Worker] rlsc-worker.ts (pg-boss.work) → AgenticRLSCStructuriserManager.process() ├─ Generator (LLM 호출, 1차 추론) ├─ Evaluator (품질 평가) └─ Feedback Loop (재추론, max iteration까지) → 성공: layouts.structured_content_agentic_jsonb 저장 rlsc_results insert (type='agentic', trace_id, batch_id) → 실패: total_status='failed_agentic_rlsc', pg-boss 재시도 → 타임아웃: reapTimeOutJobs 워커가 별도로 정리 [DB] layouts · rlsc_results · job_history · design_objects (Supabase/PostgreSQL)

문제 정의

여러 팀·서비스에서 OpenAI를 사용하면서 "어디서 얼마나 쓰는지" 파악·예산 관리가 어려웠습니다. Costs API는 계정 전체 합계만 제공해 프로젝트별 분해가 불가능했고, 별도 분석 시스템이 필요했습니다.

접근 (단계별)

  • 1) 사용 현황 전수 조사 — Default API Key 문제(이름·프로젝트 연동·비용 구분 불가) 정리, 별도 키 발급으로 추적 가능하게 설계.
  • 2) Usage API vs Costs API 비교 — Usage API + 모델별 단가 수동 계산이 프로젝트별 비용 산정의 유일한 방법임을 문서화.
  • 3) 분류 체계 — 환경 = Project / 기능 = API Key, 명명 규칙 기능-환경 (예: MDI-workflow-dev).
  • 4) 비용 테이블 스키마 정의 — start_time, input/output_tokens, project_id, api_key_id, model, input_cached_tokens, num_model_requests 등.
  • 5) Databricks Job 수집 파이프라인 — 일별 Usage API 호출, 응답 스키마 변경 이슈는 mergeSchema 옵션으로 해결.

비용 계산식

# 일반 cost = (input_tokens + output_tokens) / 1000 × model_unit_price # 캐시 적용 시 cost = (input_tokens - input_cached_tokens)/1000 × input_price + (input_cached_tokens/1000) × cached_input_price + (output_tokens/1000) × output_price
