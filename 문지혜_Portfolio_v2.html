AI 검색·생성 품질을 결정하는

LLM 기반 디자인 온톨로지 추출 Agent System 개발

Multi-Agent LLM Pipeline 14만+ pages PoC → Production

프로젝트 소개

프로젝트명: AI 프레젠테이션 자동 생성을 위한 디자인 구조 추출 시스템 구축

미리캔버스 유저의 프레젠테이션 제작 시간을 획기적으로 줄이기 위해, 사용자가 입력한 내용을 AI가 자동으로 시각화해주는 기능 개발을 위한 핵심 데이터 기반을 구축하는 프로젝트.

기존 14만+ 프레젠테이션 디자인에서 구조(레이아웃·역할·계층)를 AI가 이해할 수 있는 형태(디자인 온톨로지; R.L.S.C)로 추출하는 멀티 에이전트 시스템(Generator·Evaluator·피드백 루프)을 설계·구현.

핵심 기여

  • Generator 모듈 구현 — 디자인 → 구조 데이터 (R.L.S.C) 변환 LLM 로직 전체 구현
  • 피드백 프롬프트 설계 — R.L.S.C 생성 프롬프트 + Evaluator 평가 결과 반영
  • 대량 추출 파이프라인 설계·구현 — pg-boss 큐·워커·REST API·백오피스 UI 개발
  • 품질 검증 체계 — 데이터 샘플링 후 육안 검수 → 목표 초과 달성 (정답률 90% 달성)

문제 ①

디자인의 구조(레이아웃·역할·계층)는 복잡하고 모호해서, LLM에 한 번 요청하는 것만으로는 정확도가 목표(Page 80%, Component 95%)에 미달.

해결 ①

Generator → Evaluator → 피드백 → 재추론 반복 루프를 설계.

Evaluator가 "어떤 부분이 틀렸는지"를 판단하면, 그 결과를 피드백 프롬프트로 구성해 LLM에 다시 전달. 반복할수록 정답률이 수렴하는 구조 설계.

결과 ①

육안 검수 시행 후 목표치 달성. 이후 약 143,790 pages 전체 추출 의사결정 근거 마련.

문제 ②

14만+ 페이지를 안정적으로 처리하는 스케일 문제.

143,790 pages를 LLM으로 처리하면 수만 건의 API 호출이 발생. 실패·타임아웃·재시도를 관리하지 않으면 대량 추출이 불가능.

해결 ②

pg-boss 큐 + 워커 아키텍처 설계 및 구현.

  • 각 페이지를 비동기 job으로 등록하고, 워커가 큐에서 하나씩 꺼내 처리
  • job_history 테이블로 pending/processing/completed/failed 상태 관리
  • 타임아웃 감지 워커가 장시간 미완료 job을 자동 정리하도록 구현
  • Langfuse 연동으로 LLM 추론 이력 관리 및 LLM 호출별 비용·토큰·모델·실패 원인 추적

결과 ②

[검증] PoC 300 pages 부하 테스트로 병렬 처리 안정성 확인

  • 성공률 98%+ (295~296 / 300) — 실패 4~5건은 재시도·타임아웃 정리 워커로 자동 복구
  • 병렬도 2배 확장(60→120 병렬) 시 처리시간 43%↓ (36분 8초 → 20분 39초)

02
AI 검색·생성 정확도를 보장하는

디자인 온톨로지(R.L.S.C) 품질 검증·자동 교정 시스템 구축
— Linter 모듈 개발

Quality Gate Rule-based + Auto-fix 10,000+ 확보 목표 데이터 기반 의사결정

도입 이유

14만+ 템플릿의 디자인 온톨로지(RLSC)를 추출했지만, Page 단위 약 8%에서 오류가 발생.

문제 심각성

RLSC는 여러 하류 모듈이 의존하는 핵심 데이터:

하위 모듈RLSC 오류 시 영향
컴포넌트 검색잘못된 구조로 매칭 → 부적합 결과
내용 대치Title/Description 역할 오류 → 제목 위치에 본문 삽입
스타일 대치역할 오류 → 제목 스타일이 본문에 적용
AIP (AI Presentation)부정확한 컴포넌트로 슬라이드 생성

각 모듈에서 개별 대응하면 로직 중복·혼란이 발생하므로, RLSC 자체의 품질을 보장하는 것이 근본적 해결책으로 판단.

프로젝트 목표

내용 시각화 프로덕션 출시 전, 정확한 페이지 데이터(R.L.S.C) 10,000개 이상 확보.

  • Rule-based Linter: 오류 검출 규칙 개발
  • Auto-fix: 자동 수정 기능 개발

기여 ①
품질 지표 정의

Linter 통과율이 실사용 품질을 보장하지 못하는 맹점을 발견하고, Human labelling 기반 Accept Rate를 핵심 품질 지표로 정의·합의.

"Linter 통과 83% vs Human Accept 45%" 갭을 드러내, 팀의 목표를 '통과율'에서 'Accept Rate'로 전환.

  • Linter 통과 후 Human accept를 TP, Human reject를 FP로 정의, Human Accept Rate = TP / (TP + FP)를 Precision 지표로 관리
  • Linter 기준을 강화하면 Precision은 오르지만 실제 확보 가능한 컴포넌트 수가 줄어들 수 있으므로, TP 절대량을 Recall 성격의 가드레일 지표로 함께 모니터링

기여 ②
Rule 도입 Gate 정의

"문제를 많이 잡는 Rule = 좋은 Rule"이라는 통념을 거부하고, 정상 데이터 손실까지 함께 보는 정량 도입 기준(Gate)을 정의해 Rule 채택 의사결정을 표준화.

  • trade-off의 의사결정 규칙화: Rule이 reject를 잡는 효과(reject 제거율)와 정상 데이터를 잘못 거르는 부작용(active 제거율)을 같은 저울에 올려, reject 제거율 ≥ 2 × active 제거율 Gate를 도입 기준으로 명문화
  • 데이터 기반 채택 파이프라인 구축: 도입 전 Production 검수 데이터로 영향도를 측정해, Gate 미달 Rule은 운영에서 자동 제외 → 사람 감이 아닌 일관된 기준으로 Rule 채택 결정
  • "Rule 추가 = 품질 개선"이 성립하는 루프 완성: precision만 올리고 확보량을 깎는 Rule을 도입 단계에서 차단해, 규칙이 늘어날수록 실제 사용 가능한 컴포넌트가 함께 늘어나는 구조

기여 ③
Reject Category 체계화

흩어져 있던 Human reject 사유를 구조화하고, Category별 영향도를 정량화하여, 팀이 "무엇을 blocking하고 무엇을 무시할지" 합의할 수 있는 의사결정 근거를 마련.

  • 정성 코멘트 → 정량 의사결정 데이터화: 라벨러의 자유 서술 reject 사유를 Grouping 오류(G)·디자인 결함(D) 등 카테고리로 구조화하고 노드 단위로 태깅 → 어떤 오류가 품질을 얼마나 깎는지 측정 가능한 형태로 전환
  • 합의를 위한 근거 제공: Category별 Accept Rate 시뮬레이션으로 "특정 오류를 무시하면 목표(95%)에 얼마나 가까워지는지"를 수치로 제시 → 팀이 blocking/무시 경계를 데이터 기반으로 논의·합의할 수 있는 토대 마련

AI Data Pipeline & Ops

AI 학습데이터 확보를 위한 XML 변환 파이프라인 자동화

Docker · ECR GitHub Actions CI/CD Airflow · AWS Batch 재빌드 80%↓

프로젝트 소개

사내에 쌓인 대량 XML 디자인 데이터를 AI가 학습할 수 있는 형태(Image·CSV)로 전처리·변환하는 모듈을 개발하고, 이를 컨테이너 기반 배치 파이프라인으로 자동화해 안정적으로 학습데이터를 확보한 프로젝트.

핵심 기여
(개발 + 인프라)

  • 변환 모듈 개발: Python/Typescript 기반 XML → Image·CSV 변환 모듈(xml-image-csv-converter)을 구현하고 Docker 이미지로 패키징, AWS ECR에 배포
  • CI/CD 구축: GitHub Actions로 develop 푸시 시 빌드 → ECR 자동 업로드, AWS OIDC·IAM Role 기반 인증으로 수동 배포 제거
  • 인프라 오케스트레이션: Airflow DAG로 실행 시점·재시도·병렬을 제어하고, AWS Batch로 변환 작업을 컨테이너 Job 단위로 실행해 리소스를 자동 확장·회수

문제

  • 수동 EC2 운영: 대량 변환이 필요할 때마다 고사양 EC2(r7i.xlarge)를 최대 6대까지 직접 띄우고 끄는 수작업. 스케줄링·실패 재시도·리소스 회수가 없어 운영 부담이 크고, 작업 끝난 인스턴스가 방치되면 비용 낭비 문제 발생
  • 느리고 반복적인 빌드·배포: 이미지 빌드·푸시가 모두 수동이고, 코드가 조금만 바뀌어도 전체 레이어가 무효화돼 매번 풀 빌드 (약 15분 44초)

해결 과정

  • 변환 모듈을 컨테이너화해 실행 환경을 표준화 — 어디서든 동일하게 실행 가능
  • Dockerfile 레이어 최적화(변경 빈도 낮은 순서 배치 + requirements.txt 분리 복사) + GitHub Actions 캐시(--cache-from/to type=gha) 적용 → 빌드 단축
  • GitHub Actions로 빌드·푸시 자동화, 수동 명령 제거
  • 실행 환경을 수동 EC2 → AWS Batch로 전환하고 Airflow DAG로 일배치 스케줄링 — 사람이 인스턴스를 직접 관리할 필요 제거

결과 / 임팩트

  • 수동 EC2 운영 완전 제거 → 대량 XML 변환을 일배치로 무인 자동화, 운영 개입 최소화
  • 작업량 증가 시 자동 확장 / 종료 시 자동 회수로 리소스를 쓴 만큼만 사용 (방치 인스턴스 비용 제거)
  • 빌드 시간 단축: 15분 44초 → 13분 45초, 재빌드 시 캐시로 약 80% 단축
  • AI 학습에 필요한 전처리 데이터를 안정적·반복적으로 확보하는 기반 마련

  • 추적 가능한 키/프로파일 관리 설계: OpenAI는 Project·API Key 명명 규칙, Bedrock은 Inference Profile + 태그 체계로 기능 단위 비용 분리

AI를 활용한 템플릿 유사도 검증 자동화

DinoV2 18만 건 육안 88%↓ 2단계 검수 체계

프로젝트 소개

디자인 엔진 1.0 → 2.0 대규모 변환(약 18만 건)에서 변환 품질을 검증하기 위해, AI 이미지 유사도(DinoV2) 기반 자동 검수 파이프라인을 설계·구축한 프로젝트.

AI가 전수를 1차 판정하고, 유사하지 않다고 판단한 건만 사람이 육안검사하도록 해 검수 물량을 크게 감소.

핵심 기여

  • AI 유사도 기반 자동 검수 파이프라인 설계·개발
  • AI 1차 판정 + 의심 건만 육안검사 2단계 체계 도입
  • 유사도 모델·임계값 검증·튜닝 (DinoV2 채택)
  • 18만 건 배치 안정화·병렬처리 및 운영 자동화

문제

  • 디자인 1.0 → 2.0 변환 후, 2.0 썸네일이 원본과 일치하는지 18만 건을 알바 인력이 전수 육안검사
  • 주 1~2회 반복엔 인력·시간 부담이 크고, 사람마다 판단 기준이 달라 일관성 확보도 어려움

가설 설정

  • 1.0 원본 vs 2.0 썸네일을 이미지 유사도로 정량화하고 임계값으로 분류하면,
  • AI가 전수를 1차 판정 → 임계값 이상(유사)은 자동 통과, 미만(유사하지 않음)만 사람이 육안검사

→ 육안검사를 의심 건으로만 한정해 검수 리소스를 줄일 수 있다.

해결 과정

1) End-to-end 자동 검수 파이프라인 설계 및 구축

2) 2단계 검수 체계 설계 — AI 유사도 점수로 1차 전수 자동 검수 → 임계값 이상은 자동 통과, 미만으로 판정된 건만 사람 육안검사 대상으로 라우팅

3) 유사도 모델 검증·임계값 튜닝

  • 육안 라벨을 정답으로 한 혼동행렬(TP/FP/TN/FN)·400개 데이터셋 구성
  • 이미지를 grid로 쪼개 각 그리드 유사도의 최소값으로 전체 유사도를 판정하는 방식 설계, grid 5~25 · threshold 실험
  • DinoV2 vs DreamSim 비교 → DreamSim이 정확도는 약간 높지만 DinoV2가 처리 속도가 빨라 채택
  • 최종 grid=13 / threshold=0.72 (검증셋 32개 중 오탐 1개)로 판정 기준 확정

4) 대규모 배치 안정화·병렬처리

  • Puppeteer protocol timeout 방지를 위한 500건 단위 브라우저 재시작
  • Node 프로세스 2개 병렬화: Process 1(합성 요청·CompositeId 수신) / Process 2(SQS 수신·DynamoDB 매칭·output S3)
  • 커스텀 에러 로그(converter·sqs·canvas 에러 분류)로 대량 처리 디버깅·안정성 확보

결과 / 임팩트

  • 육안검사 물량 축소: 모델 개선 후 유사하다고 판단한 템플릿 수가 약 5.7배로 증가 → 육안 검사 대상 88% 감소
  • 판정 신뢰도 확보: 검증셋 기준 오탐 1/32 수준의 임계값을 정량 도출해, AI 자동 검수의 근거를 데이터로 확보
  • 일관성 향상: 사람마다 다르던 판단을 정량 유사도 기준으로 표준화
  • 처리량·운영 효율: 18만 건을 주 1회 처리 가능한 자동 배치 기반 마련
