# OpenAI 비용 대시보드 구축

> 조직 내 **OpenAI API 사용량·비용**을 **프로젝트·API Key·모델** 단위로 수집·집계해 대시보드를 구축한 설계·구현 내역.  
> **출처**: Confluence MGRSERVER [AI 비용 대시보드 폴더](https://miridih.atlassian.net/wiki/spaces/MGRSERVER/folder/1203307187) — OpenAI 비용 대시보드 구축 방안, 비용 테이블 스키마, 분류 체계, 사용 현황 조사, OpenAI API DataBricks Job

---

## 1. 개요·목표

| 구분 | 내용 |
|------|------|
| **목적** | OpenAI API **비용·사용량**을 **프로그램적으로 수집**하고, **프로젝트·API Key·모델** 단위로 분해해 대시보드로 가시화 |
| **배경** | 여러 팀·서비스에서 OpenAI를 사용하면서 **어디서 얼마나 쓰는지** 파악·예산 관리 필요. Costs API만으로는 계정 전체 합계만 제공되어 **프로젝트별 분해 불가** |
| **결과** | Usage API 기반 **비용 테이블 스키마** 정의, **DataBricks Job**으로 수집 파이프라인 구축, API 키 관리·분류 체계(가안) 수립, 사용 현황 전수 조사 |

---

## 2. 빌드업 (어떤 점이 어떻게 됐는지)

| 단계 | 내용 |
|------|------|
| **1) 조사** | **OpenAI 사용 현황 전수 조사** — Default API Key 문제(이름·프로젝트 연동 불가, 비용 구분 불가, Databricks 추적 불편) 정리. **별도 API Key 발급**으로 키별 비용 분리·추적 가능하도록 방향 설정. |
| **2) API 정리** | **Usage API vs Costs API** 비교. **Usage API + 모델별 단가 수동 계산**이 **프로젝트별 비용 계산의 유일한 방법**임을 문서화. Costs API는 계정 전체·일자별 합계만 제공. |
| **3) 구축 방안** | **OpenAI 비용 대시보드 구축 방안** 문서 작성 — 엔드포인트, curl 예시, **Project 중심 관리**(사용량/비용 분해, 예산 알림), API 키 관리 현황(스프레드시트), Admin Key·Databricks Secret Scope 등 전제 조건 정리. |
| **4) 분류 체계** | **환경 단위 = Project**, **기능 단위 = API Key**, Key 이름 규칙(기능-환경, 예: MDI-workflow-dev, AI-tool-prod). Usage API의 **model 필드**로 Key 하나로도 모델별 비용 집계 가능함 정리. |
| **5) 스키마·비용 계산** | **Usage API 응답 기준 비용 테이블 스키마** 정의(start_time, input_tokens, output_tokens, project_id, api_key_id, model 등). **비용 = (input_tokens + output_tokens)/1000 × 모델별 단가**, **input_cached_tokens** 반영(캐시 할인 단가) 수식 문서화. |
| **6) 수집 파이프라인** | **DataBricks Job**으로 Usage API 호출·수집. **OpenAI API 응답값 변경**으로 스키마 불일치 시 Job 실패 → **mergeSchema 옵션**으로 해결. |

---

## 3. 내가 한 작업 (역할)

- **OpenAI 사용 현황 조사**: Default API Key 특성·문제점 정리, 별도 API Key 발급 해결책, 전수 조사 스프레드시트·관련 문서 링크 정리.
- **OpenAI 비용 대시보드 구축 방안** 문서 작성: Usage API / Costs API 역할, 엔드포인트·curl 예시, Project·API Key 관리 방침, Admin Key·Databricks Secret 전제 조건, 도움 요청 사항 정리.
- **OpenAI 분류 체계 (가안)** 제안: 환경=Project, 기능=API Key, Key 명명 규칙, 모델별 집계 가능성 설명.
- **OpenAI API 비용 테이블 스키마 (Using API)** 정의: Usage API 응답 기준 컬럼 정의(start_time, end_time, input_tokens, output_tokens, project_id, api_key_id, model, input_cached_tokens 등), 비용 계산식, 모델별 단가·캐시 할인 반영, Google 시트(미리클>AI도구 비용 계산) 링크.
- **OpenAI API DataBricks Job**: Usage API 데이터 수집 Job 구현·운영. **응답 스키마 변경**으로 인한 Job 실패 시 **mergeSchema 옵션** 적용해 스키마 불일치 해결.

## 4. 사용한 툴·기술

| 구분 | 툴·기술 |
|------|----------|
| **API** | OpenAI **Usage API** (GET /v1/usage), **Costs API** (GET /v1/usage/costs) — total check용 |
| **수집·저장** | **DataBricks** (Job, Secret Scope에 API Key 저장), 테이블 스키마 |
| **비용 계산** | 모델별 단가(OpenAI Pricing), input/output/cached 토큰 수, 수식 적용 |
| **관리·문서** | Google 스프레드시트(API 키 현황, 미리클>AI도구 비용 계산), Confluence MGRSERVER |

---

## 5. 워크플로우

### 5.1. 데이터 수집 (개념)

1. **Admin API Key**로 조직 전체 Usage API 호출 (일반 키는 해당 키 요청만 조회 가능).
2. **기간·granularity** (분/시/일), **project_id·model** 등 필터로 사용량 조회.
3. DataBricks Job에서 **정기 실행** (예: 일별)으로 Usage API 호출 → **비용 테이블 스키마**에 맞게 적재.
4. **mergeSchema** 옵션으로 OpenAI 응답 스키마 변경 시에도 기존 테이블에 컬럼 추가·적재 가능하도록 처리.

### 5.2. 비용 계산

- **비용** = (input_tokens + output_tokens) / 1000 × **모델별 단가($)**  
- **캐시 적용 시**:  
  `(input_tokens - input_cached_tokens)/1000 × input_price + (input_cached_tokens/1000) × cached_input_price + output_tokens/1000 × output_price`

### 5.3. 대시보드

- 적재된 테이블을 **프로젝트·API Key·모델·일자** 기준으로 집계해 **대시보드**(시각화 도구는 팀 내 사용 툴에 따름)에서 조회.

## 6. 비용 테이블 스키마 요약 (Usage API 기준)

| 컬럼 | 설명 |
|------|------|
| start_time, end_time | UTC 집계 구간 (Unix timestamp) |
| start_time_kst, end_time_kst | KST 변환값 |
| object | 리소스 타입 (예: organization.usage.completions.result) |
| input_tokens, output_tokens | 토큰 수 |
| num_model_requests | API 호출 횟수 |
| project_id, api_key_id, user_id | 프로젝트·키·유저 |
| model | 모델명 (예: gpt-4o-2024-08-06) |
| input_cached_tokens | 캐시된 입력 토큰 (할인 적용 분) |
| input_audio_tokens, output_audio_tokens | 음성 모델용 (Whisper 등) |
| project_name | 수동 매핑 프로젝트명 (예: miridih-slack) |

---

## 7. API 키·분류 체계 요약

- **환경 단위 = Project**: 운영/개발/연구 등 Project로 분리, 예산·제한을 환경별 관리.
- **기능 단위 = API Key**: 같은 Project 내에서 기능·서비스별 Key 발급, **이름 규칙**: `기능-환경` (예: MDI-workflow-dev, AI-tool-prod).
- **모델별 관리**: 한 Key로 여러 모델 사용해도 **Usage API의 model 필드**로 모델별 사용량·비용 집계 가능.
