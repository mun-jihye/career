## 2. 빌드업 (흐름 관점)

- **1.0 디자인(xml)** 기반 데이터가 **2.0 디자인(json)** 으로 전환되면서,  
  “변환 → 검증(이미지 비교·유사도·육안검사) → 통과분 DB 저장 → 합성” 단계가 필요해짐.
- **초기**: 수동 또는 단발 실행 위주.
- **이후**: 각 단계별 **배치 프로세스**로 정리해 **재실행·스케줄·에러 대응**이 가능하도록 구성.
- **결과**: 템플릿 재저장, 컨버터 에러 수정, 정합성(이미지 비교·육안검사 시트), DB 저장, 합성기 포맷별 요청까지 **일련의 배치**로 운영.

## 3. 내가 한 작업 (역할)

- 위 5종 배치에 대해 **요구사항 정리·실행 흐름 설계·구현·운영** (스크립트/서비스 작성, 실행 조건·입출력·에러 처리 정리).
- **AI-599**: 이미지 비교·유사도 점수·육안검사 시트 생성 배치 개발.

## 5. 워크플로우 (개념)

1. **템플릿 재저장**: (조건 만족 시) 재저장 API 호출 배치 실행.
2. **컨버터 에러 대응**: 템플릿·조합요소·svg 단위로 1.0→2.0 변환 재시도 또는 에러 수집·처리 배치.
3. **이미지 비교·육안검사**: AI 모델로 이미지 비교·유사도 점수 산출 → 육안검사 시트 생성 배치 (AI-599).
4. **DB 저장**: 육안검사 통과한 json에 대해 DB 저장 API 배치 실행.
5. **합성**: 합성기 에러 대응을 위해 포맷별(webp, pptx, pdf, mp4, gif)로 합성 요청 배치 실행.

(각 배치의 상세 트리거·스케줄·입출력은 Confluence·코드베이스 참고.)
