  • 기획서 없이 스스로 정의: 누구도 시키지 않은 문제를 직접 발견 → "여러 팀이 OpenAI 쓰는데 비용 관리가 안 되네" 라는 인사이트에서 출발
  • 도구 한계 식별 → 대안 설계: OpenAI Costs API는 계정 합계만 제공 → 프로젝트별 분해 불가 → Usage API + 자체 단가 계산이 유일한 방법임을 문서화
  • 분류 체계 거버넌스: 환경=Project, 기능=API Key, 이름 규칙(`기능-환경`, 예: `MDI-workflow-dev`) — 향후 다른 팀이 새 키를 만들 때도 자동으로 분류되는 시스템
  • 장애 대응 정책: OpenAI 응답 스키마 변경 시 Job 실패 → mergeSchema 옵션으로 자동 적응

Why Product Manager?

엔지니어로서 쌓은 경험이 PM 직무에서 어떻게 차별화된 강점이 되는지 정리했습니다.

1

기술 실현 가능성 판단

엔지니어 출신이라 "이게 가능한가/얼마나 걸리는가"를 본능적으로 가늠 → 비현실적 요구사항을 거르고 현실적 로드맵을 만듭니다.

2

AI Product 도메인 전문성

LLM Agent · 프롬프트 설계 · 데이터 파이프라인 · 품질 지표를 직접 다뤄봐서 AI 제품 PM의 진입 장벽을 이미 통과한 상태입니다.

3

운영 PM 감각

14만+ 스케일, 배치 5종, 비용 거버넌스 등 실제 운영 중인 시스템을 책임져본 경험. PoC만이 아니라 launching 후 안정화까지 책임집니다.

4

문서·커뮤니케이션

Confluence 기반 기획 문서를 평소에 직접 써왔고, "왜 이걸 하는가"부터 명확하게 정렬합니다. 비개발자에게도 기술적 의사결정 근거를 잘 풀어냅니다.
